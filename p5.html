Reference · Generative AI Use Case 04

★ メール返信スキル 設計ガイド

生成AIの「メール返信スキル」を設計・改善するための設計思想・作成手引き・注意事項をまとめた参考ドキュメントです。

Section 02

description フィールドの設計指針

最重要。AIはここだけを見てスキルを使うか判断します。

原則

1 「いつ使うか」と「何をするか」を両方書く
2 やや積極的（pushy）に書く → 使われなさすぎを防ぐため
3 100語以内が目安

現在の設定例

仕事のメール文面を貼り付けると、ビジネス日本語の返信案を自動生成するスキル。 受信メールの文脈・トーン・緊急度を読み取り、適切な敬語レベルで返信を構成する。 「このメールに返信して」「返信案を作って」「メールの返信を書いて」など、 メール返信の下書き作成を求められたら必ずこのスキルを使うこと。 ユーザーがメール文面を貼り付けた場合も積極的にこのスキルを適用すること。

良い description の特徴

✓ 良いトリガーとなるユーザーフレーズを複数列挙している
✓ 良い「必ずこのスキルを使うこと」など明示的な指示が入っている
✓ 良い曖昧な条件（「メール文面を貼り付けた場合も」）をカバーしている

避けるべき description

✕ NG「〜に使えるスキル」のような受け身表現のみ
✕ NGトリガーフレーズの列挙なし
✕ NG機能説明のみで「いつ使うか」が不明
Section 03

本文（スキル本体）の設計原則

3-1. 出力フォーマットを先に定義する

AIは出力フォーマットが明示されていると品質が安定します。メール返信であれば以下の順序を固定し、プレースホルダー形式（〇〇様、【要確認】）を明示します。

宛名 → 挨拶 → 本文 → 結語 → 署名欄

3-2. 判断ルールをテーブル化する

「社外 vs 社内」「初回 vs 継続」などの条件分岐は、文章よりテーブル形式がAIに伝わりやすいです。

受信メールの特徴 使用する文体
初めての取引先 最上級の敬語（〜でございます）
継続取引先 標準的なビジネス敬語

3-3. 実行手順をステップ番号で書く

順序を明確にすることで漏れが減ります。

1メール解析
2意図把握
3文体判断
4返信生成
5確認ポイント提示

3-4. 「要確認」ルールを階層化する

情報の優先順位を明示しておくと、不必要な【要確認】が減り、すぐ使える返信案になります。

ユーザー提供情報 ＞ メール本文から推定 ＞ 不明（→ 【要確認】）
Section 06

拡張ポイント（今後の改善候補）

件名の自動生成　返信メールの件名（Re: 〇〇）も出力する
英語メール対応　英語での受信メールに対して英語返信を生成
テンプレートバリエーション　お礼・謝罪・提案など用途別テンプレートを追加
署名の自動保存　ユーザーの署名情報をメモリに記憶して毎回自動補完
Section 07

テスト観点チェックリスト

スキル更新後は以下のシナリオで動作確認します。

社外・初回取引先のメールに対して → 最上級の敬語が使われるか
社外・継続取引先のメールに対して → 自然なです・ます調になるか
社内・同僚からのメールに対して → 過剰な敬語が出ないか
複数の質問を含むメールに対して → 番号付きで漏れなく回答するか
日付・金額が不明な場合 → 【要確認】が適切に表示されるか
ユーザーが署名情報を事前提供 → 署名欄に自動反映されるか
「返信案を作って」とだけ言った場合 → スキルがトリガーされるか
メール文面を貼り付けただけの場合 → スキルが積極的に適用されるか
